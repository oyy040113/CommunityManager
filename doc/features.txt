2	相关技术概述
本章将详细介绍学生社团信息整理系统开发涉及的核心技术，包括后端框架 Spring Boot、前端框架 Vue、持久层框架 MyBatis、关系型数据库 MySQL、权限验证技术 JWT 及文件存储技术，这些技术协同构成系统的技术基石，支撑系统高效稳定运行。
2.1	SpringBoot框架
SpringBootSpring Boot 是 Pivotal 团队基于 Spring 框架开发的开源后端框架，核心优势在于 “约定优于配置”，大幅简化 Java 企业级应用的开发流程[20]。它通过自动配置机制，将传统 Spring 框架中 80% 以上的 XML 配置转化为自动化处理，开发者仅需引入对应场景的 Starter 依赖（如spring-boot-starter-web），即可快速激活嵌入式 Tomcat 服务器、RESTful 接口支持等核心能力，无需手动配置数据源、DispatcherServlet 等组件[21]。
2.1.1 核心特性与生态
1)自动配置：根据项目依赖和环境变量自动适配配置，减少重复编码，其本质是对 Spring 开发技术的升级，通过引入注解简化了 Web 框架的配置和开发流程[14]
2)嵌入式服务器：内置 Tomcat、Jetty 等服务器，支持项目打包为可执行 JAR 包，直接部署运行，无需额外部署 WAR 文件[13]；
3)生产级监控：通过 Actuator 模块提供健康检查、性能指标监控等功能，便于运维管理，符合生产级应用的核心需求[20]；
4)Starter 机制：提供标准化依赖组合（如spring-boot-starter-mybatis），快速集成第三方框架，实现依赖包和版本的全自动管理[18]；
5)核心生态兼容：完美支持 Spring MVC、Spring Security 等 Spring 生态组件，同时兼容 MyBatis、Redis 等主流工具，基于控制反转 IOC 和面向切面编程 AOP 的设计，可通过 @Autowired 注解实现对象的依赖注入[18]。
2.1.2 框架架构与启动流程
Spring Boot 采用分层架构设计，与 MVC 模式深度契合，其架构分为三层核心模块[8]：
1)	控制层（Controller）：接收前端 HTTP 请求，进行参数校验与响应封装；
2)	服务层（Service）：封装核心业务逻辑（如社团审批、活动报名校验），实现模块间解耦；
3)	数据访问层（Mapper）：通过 MyBatis 等持久层框架与数据库交互，执行数据 CRUD 操作。
启动流程通过主程序的@SpringBootApplication注解触发，依次完成环境准备、Spring 容器初始化、Bean 加载、应用启动等步骤，最终实现项目快速部署与运行[20]。Spring Boot 框架架构图如下：

2.2	Vue框架
Vue 是一套渐进式 JavaScript 前端框架，以 “组件化开发” 和 “响应式数据绑定” 为核心，适配从简单页面到复杂单页应用（SPA）的全场景开发需求[16]。与传统前端开发模式相比，Vue 通过 MVVM（Model-View-ViewModel）架构实现数据与视图的解耦，大幅提升开发效率与代码可维护性，尤其适合学生社团系统中社团列表、活动报名、数据统计等交互密集型场景[5]。
2.2.1 核心特性与技术原理
1)响应式数据绑定：通过数据劫持（Object.defineProperty 或 Proxy）与虚拟 DOM 技术，实现数据变化时视图自动更新，无需手动操作 DOM[16]；
2)组件化开发：将页面拆分为独立可复用的组件（如社团卡片、活动表单、统计图表），每个组件包含模板（Template）、逻辑（Script）、样式（Style），支持跨页面复用，代码复用率可达 75%[20]；
3)路由管理：依托 Vue Router 实现 SPA 页面无刷新跳转，通过路由守卫控制页面访问权限（如未登录用户禁止进入社团管理页面）；
4)状态管理：通过 Pinia（Vue3 推荐）管理全局共享数据（如用户登录状态、社团分类列表），解决组件间通信难题[14]；
5)生态工具链：Vue CLI 提供项目快速初始化能力，Element Plus 组件库可直接复用表单、表格、弹窗等常用 UI 组件，降低界面开发成本，前端通过 axios 的 post 方法与后端服务器进行数据交互[14]。
2.2.2 MVVM 架构原理
MVVM 架构分为三层，各层职责明确且通过数据绑定实现联动[16]：
1)模型层（Model）：存储业务数据（如社团信息、活动数据），对应后端返回的 JSON 数据或本地存储数据；
2)视图层（View）：用户交互界面（如社团列表页、活动报名页），通过模板语法绑定 Model 数据；
3)视图模型层（ViewModel）：作为中间桥梁，监听 Model 数据变化并同步至 View，同时捕获 View 的用户操作并更新 Model，实现 “数据驱动视图”。
这种架构设计使业务逻辑与 UI 展示分离，ViewModel 将 Controller 中数据展示的业务逻辑抽取出来，避免了传统 MVC 模式中 View 驱动过于频繁的问题[18]。例如学生报名社团时，用户点击 “报名” 按钮（View 操作）→ViewModel 接收事件并调用后端接口→Model 数据更新（报名状态变更）→View 自动刷新（显示 “已报名” 状态），全程无需手动操作 DOM[4]。Vue MVVM 架构原理图如下：

2.3	MyBatis 框架
MyBatis 是一款轻量级 Java 持久层框架，通过封装 JDBC 底层操作，实现数据库交互的简化与优化[15]。它支持 XML 或注解两种方式管理 SQL 语句，将数据访问逻辑与业务逻辑分离，既保留了 SQL 语句的灵活性，又避免了原生 JDBC 的重复编码工作，特别适合学生社团系统中多条件查询、复杂关联查询等场景[20]。
2.3.1 核心优势
1)灵活的 SQL 管理：通过 XML 文件集中配置 SQL，支持动态 SQL（如if、choose标签），适配多条件查询（如按社团名称、分类筛选社团）、动态更新等复杂场景，部分复杂 SQL 语句可通过原生态 SQL 实现[14]；
2)半自动化 ORM 映射：通过resultMap实现数据库字段与 Java 实体类属性的精准映射，支持一对一（如社团与指导老师）、一对多（如社团与成员）等复杂关联查询；
3)轻量级设计：核心包体积小，学习成本低，无需依赖复杂容器，可快速集成到 Spring Boot 项目中；
4)缓存机制：内置一级缓存（SqlSession 级别）和二级缓存（Mapper 级别），减少重复数据库查询，提升系统性能，例如频繁访问的社团列表数据可通过缓存快速响应[20]。
2.3.2 工作流程
1)开发者通过 XML 或注解定义 SQL 语句与映射规则（如将club表字段映射到Club实体类）[15]；
2)系统启动时，MyBatis 加载映射文件并解析 SQL 配置[20]；
3)业务层通过 SqlSessionFactory 获取 SqlSession 实例[20]；
4)SqlSession 调用 Mapper 接口方法（如ClubMapper.selectByCondition），执行对应的 SQL 语句[15]；
5)MyBatis 将数据库返回结果映射为 Java 实体类，返回给业务层[14]。
2.4	MySQL 数据库
MySQL 是一款开源关系型数据库，以高性能、高可靠性、易扩展性为特点，广泛应用于 Web 应用开发。它支持 InnoDB 存储引擎，提供事务 ACID 特性、行级锁、外键约束等核心功能，能够满足学生社团系统中数据存储、查询、事务处理等需求，例如社团注册、活动报名等操作需通过事务保证数据一致性[2]。
2.4.1 核心特性与设计原则
1)事务支持：支持 ACID 特性，确保多步操作的原子性（如社团成员加入时，同时更新成员表与社团人数统计）[2]；
2)索引优化：支持 B + 树索引，针对频繁查询的字段（如社团 ID、用户学号）建立索引，提升查询效率；
3)数据完整性：通过主键、外键、非空约束等保证数据准确性，例如社团表的advisor_id关联指导老师表的id，确保指导老师存在[19]；
4)扩展性强：支持主从复制、分库分表，可根据系统用户规模扩展存储能力。
2.4.2 在系统中的应用
MySQL 用于存储系统所有核心数据，包括用户信息（学生、社团负责人、管理员）、社团信息（名称、分类、简介）、活动信息（主题、时间、地点）、成员关系、资源申请记录等[2]。通过 MyBatis 框架与后端业务层实现数据交互，为社团管理、活动组织、数据统计等功能提供稳定的数据支撑，其逻辑设计充分参考了同类校园管理系统的数据库设计经验[6]。
2.5	JWT权限验证
JWT（JSON Web Token）是一种轻量级的身份认证协议，基于 JSON 格式实现用户身份信息的安全传输。它采用无状态设计，无需在服务端存储会话信息，适配前后端分离架构与分布式系统，是学生社团系统权限控制的核心技术，可实现多角色（超级管理员、社团管理员、指导老师、学生）的精细化权限管控[18]。消除服务端会话存储需求的方式支持JWT得以实现跨域认证和微服务架构的适配，尤其适配前后端分离项目的权限控制需求。其自包含特性允许前端直接解析载荷获取用户信息保证加密签名保障传输过程的安全性，典型应用场景包括单点登录、API鉴权和移动端身份核验等。
2.5.1 结构组成
JWT 由三部分组成，通过.分隔，均采用 Base64 编码：
1)头部（Header）：声明令牌类型（JWT）和加密算法（如 HS256）；
2)载荷（Payload）：存储核心信息（如用户 ID、角色类型、令牌有效期），支持自定义字段（如社团 ID）；
3)签名（Signature）：通过服务端密钥对头部和载荷进行加密，确保令牌不被篡改，验证令牌完整性。
2.5.2 认证流程
1)用户登录时，后端验证账号密码通过后，生成包含用户 ID、角色的 JWT 令牌；
2)前端接收令牌并存储（如 localStorage），后续请求时在 HTTP 请求头中携带令牌；
3)后端通过拦截器验证令牌的有效性（签名是否正确、是否过期）；
4)验证通过后，解析令牌中的用户信息，根据角色权限判断是否允许访问目标接口（如仅超级管理员可审批社团注册）；
5)验证失败则返回权限错误，拒绝请求，若 access token 过期可通过 refresh token 重新获取[17]。
2.6	文件存储技术
系统需支持活动海报、策划方案、社团风采照片等多类型文件的上传、下载与预览，采用 “本地存储 + 路径映射” 的方案实现文件管理，兼顾存储效率与访问安全性。
2.6.1 核心实现
1)前端通过 Vue 的upload组件实现文件上传，支持文件格式校验（如仅允许 PNG、JPG 图片，PDF、DOCX 文档）、大小限制（如单文件不超过 10MB）[14]；
2)后端接收文件后，存储到服务器指定目录（如/upload/club/202601/），同时将文件路径存入 MySQL 数据库，关联对应的社团或活动；
3)通过 Spring Boot 配置静态资源访问路径，实现文件的直接访问与预览（如通过http://localhost:8080/upload/club/202601/poster.png访问社团海报）[2]；
4)支持文件下载功能，后端通过流的方式读取文件并响应给前端，实现活动策划案、财务报表等文件的导出。
2.6.2 安全控制
1)文件格式白名单：仅允许常见的图片、文档格式，拒绝可执行文件（如.exe），防止恶意文件上传；
2)访问权限控制：通过拦截器验证用户身份，仅允许社团成员或管理员访问对应社团的文件；
3)文件名加密：上传时对文件名进行 MD5 加密，避免文件名重复覆盖，同时隐藏原始文件名，提升安全性[14]。















3	需求分析
3.1	业务需求概述
本项目聚焦高校学生社团管理的实际痛点，旨在搭建一套基于 Vue+Spring Boot 的学生社团信息整理系统，通过数字化手段打通社团管理全流程。系统采用前后端分离架构，后端依托 Spring Boot 框架搭建核心服务，基于 MVC 三层架构设计业务逻辑，控制层负责请求分发与数据处理，服务层承接核心业务逻辑，数据层通过 MyBatis 对接 MySQL 数据库，完成数据存储与交互；前端以 Vue 框架为基础，结合 Vue Router 实现动态路由管控，集成 Element Plus 组件库优化界面交互，通过 Axios 实现前后端异步数据通信，确保信息实时同步。
系统核心由后台管理模块与前端用户模块构成，后台采用 MySQL 与 Redis 双存储方案，缓存关键数据以提升管理操作效率；前端涵盖 Web 端与移动端适配，支持社团成员、负责人、指导老师、管理员等多角色便捷操作。通过模块化开发与分层设计，系统既保障核心功能的完整性，又具备灵活的可维护性，适配高校社团规模扩张与业务场景拓展的需求，为社团管理提供高效、可行的数字化解决方案[19]。
3.2	系统功能性需求分析
3.2.1	系统整体需求分析
系统用户划分为四大核心角色，不同角色权限与功能诉求各有侧重，通过角色与功能的精准匹配，实现社团管理的规范化与高效化。
普通用户（学生）核心需求围绕社团参与展开，需支持校园统一身份认证登录（学号 + 密码），首次登录需完善个人信息（姓名、联系方式、兴趣标签、所属院系年级等）；可浏览社团分类列表、社团风采展示、活动信息，筛选感兴趣的社团与活动并在线报名；查看个人参与记录（已加入社团、已报名活动、活动参与状态）、接收系统通知与社团内部消息；活动结束后提交反馈意见与评价，对社团进行评分留言。
社团管理员（负责人）需具备社团信息维护功能，包括编辑社团简介、更新风采照片、上传活动成果、管理社团标签；成员招募与考勤管理，支持审核学生入社申请、登记成员考勤情况、统计成员活跃度；活动全流程策划与报备，涵盖创建活动、设置活动参数、上传策划案与海报、提交报备申请；经费与物资管理，记录会费收入、活动支出、赞助款项，提交场地与器材使用申请；发布社团内部通知，查看成员反馈与活动评价，统计活动参与效果与社团运营数据。
指导老师需实时掌握所指导社团的动态，包括查看社团基本信息、成员构成、活动计划与历史活动记录；在线审核社团提交的活动方案，填写审核意见（通过、驳回并说明原因）；提供指导意见，接收社团管理员发起的咨询请求，实时沟通并记录指导日志；评估社团运营状况，查看成员活跃度统计与活动效果分析，提出改进建议。
超级管理员（校社团管理部门）负责全局管控，涵盖社团注册审批（查看资质材料、批准或驳回申请）、活动报备审核（在线审核活动方案、反馈修改意见）、用户账号管理（审核注册信息、分配角色权限、启用或禁用违规账号）；资源调度分配，协调场地与器材使用冲突，确保资源合理利用；数据统计分析，生成全校社团运营数据报表；发布全校性通知（政策文件、评优结果等），监管各社团经费使用合规性。

3.2.2	后台管理模块需求分析
后台管理模块面向超级管理员与社团管理员，提供精细化的数据管理与业务审批功能，核心模块围绕权限管控、社团运营、活动统筹、资源调度等核心场景设计，各模块协同支撑管理效率提升。

（1）用户与权限管理
支持超级管理员对全系统用户账号进行全生命周期管理，包括审核学生注册信息（校验学号、院系等信息真实性）、启用或禁用违规账号、分配角色权限（普通学生、社团管理员、指导老师）等操作。系统基于 RBAC 模型实现权限分级，超级管理员拥有最高权限，可配置不同角色的访问权限与操作范围；社团管理员仅能管理本社团相关数据，指导老师仅能查看所指导社团的相关信息，确保数据安全与操作规范。
用户信息管理页面以表格形式展示账号、姓名、角色、所属院系、联系方式、注册时间、账号状态等核心字段，支持按账号、姓名、角色、院系等条件精准查询，以及信息编辑、账号冻结 / 解冻、账号删除等操作；支持导出用户数据为 Excel 格式，便于存档统计。
（2）社团管理
超级管理员可审核新社团注册申请，查看社团提交的资质材料（社团章程、发起人信息、指导老师证明等），批准或驳回注册请求并填写详细审核意见；支持对已注册社团进行分类管理（学术科研、文化艺术、体育竞技、志愿服务等）、星级评定（基于成员活跃度、活动质量、合规性等指标）、评优申报审核（接收社团评优申请、审核相关材料、公示结果）等操作。
社团管理员可维护社团基础信息，包括编辑社团简介、更新风采照片与宣传视频、上传活动成果（新闻稿、照片、视频）、管理社团标签与所属分类；可提交社团注销备案申请，说明注销原因并上传相关材料，经超级管理员审核后完成注销流程。后台需展示所有社团的运营数据，包括成员规模、活动场次、活跃度、星级评级、经费余额等，支持按分类、星级、活跃度等条件筛选查询，方便管理员全局把控。
（3）活动管理
活动管理覆盖策划、报备、执行、归档全流程。社团管理员可发布活动信息，设置活动主题、时间、地点、报名条件（年级、专业、技能要求等）、人数上限、报名起止时间等参数，上传活动策划案（PDF、DOCX 格式）与海报（PNG、JPG 格式）；提交活动报备申请后，超级管理员可在线审核，反馈修改意见（如活动时间冲突、策划案不完善等）。
活动执行过程中，支持报名人数实时统计、在线签到考勤（可通过学号、二维码扫描签到）、紧急通知发布（针对已报名成员）；活动结束后，社团管理员可上传活动总结（文字、照片、视频）、成果资料，完成归档操作；后台需保留完整活动记录，包括活动参数、报名名单、考勤数据、活动总结、成员反馈等，供后续查询与统计。
超级管理员可查看全校所有活动的报备情况与执行状态，对违规活动进行干预（暂停、取消），统计各类型活动的举办场次与参与率。
（4）资源与财务管理
资源管理模块支持场地（会议室、体育馆、报告厅等）、器材（音响、投影仪、体育器材等）等校园资源的在线申请与调度。超级管理员可维护资源信息（名称、位置、规格、可使用时段、容量），查看资源占用情况，协调资源冲突（如同一时段多个社团申请同一场地），审批资源使用申请；社团管理员可提交资源使用申请，选择资源类型、使用时间、使用用途，上传相关证明材料，跟踪审批进度，记录资源借用与归还情况（器材类）。
财务管理需支持社团经费收支明细记录，包括会费收入（记录缴费成员、金额、时间）、活动支出（场地租赁、物料采购、宣传费用等，需上传支出凭证）、赞助款项（记录赞助方、金额、使用用途）；支持财务数据导出（Excel 格式）与核对，社团管理员可按月 / 学期生成财务报表；超级管理员可查看各社团财务状况，监督经费使用合规性，对异常支出进行核查。
（5）数据统计与通知管理
数据统计模块自动采集社团数量、成员规模、活动场次、参与率、成员活跃度、经费收支等核心指标，生成可视化报表（柱状图、折线图、饼图）与排行榜（社团活跃度排行、活动参与率排行、星级社团排行），支持按时间维度（周、月、学期、学年）筛选查询，数据可导出为 Excel 格式供存档分析。
通知公告模块支持超级管理员发布全校性政策文件、评优结果、活动通知等，设置通知优先级（普通、重要、紧急）与有效期限；社团管理员发布本社团内部通知（活动提醒、会议通知、成员通知等）；后台可查看通知阅读状态（已读人数、未读人数、阅读名单），确保重要信息有效触达；支持通知编辑、删除、置顶操作，历史通知可归档保存供查询。
3.2.3	前端用户模块需求分析
前端用户模块面向学生、指导老师等角色，聚焦操作便捷性与交互友好性，核心功能模块围绕 “参与 - 管理 - 互动” 三大场景设计，适配移动端与 PC 端的多终端使用需求。

支持在线审核社团提交的活动方案，查看活动策划案、海报、报名条件等信息，填写审核意见（通过、驳回并说明原因，如活动流程不完善、安全预案缺失等），审核结果实时通知社团管理员。
可查看社团成员活跃度统计（参与活动次数、签到率、发言频率等），对低活跃度社团提出改进建议；接收社团管理员发起的咨询请求（如活动策划疑问、资源申请咨询），通过内置聊天功能实时沟通，记录指导日志（指导时间、内容、建议），形成完整的指导记录供后续查阅。
3.3	系统非功能性需求分析
3.3.1	系统性能需求
系统需适配高校师生的并发访问场景，尤其是新生入学季、社团招新期、活动报名高峰期等流量高峰，需支持至少 500 人同时在线操作，页面加载时间不超过 2 秒，数据查询响应时间不超过 500 毫秒。
通过 Redis 缓存热门数据（如社团列表、活动信息、用户权限信息），减少数据库查询压力；MySQL 数据库需建立合理索引（如用户学号、社团 ID、活动时间、报名状态等字段），优化查询效率；采用分页加载机制处理大量数据展示（如社团列表、活动列表、成员列表），避免页面卡顿。
系统需保障 7×24 小时稳定运行，年故障率不超过 0.5%，故障恢复时间不超过 30 分钟；后台服务需支持负载均衡，可根据用户规模与访问量动态扩展服务器资源，适配高校社团数量与用户规模的增长（如新增社团、学生人数增加）。
3.3.2	系统安全性需求
系统需构建多层安全防护机制，保障用户信息与数据安全。用户登录采用 JWT 权限验证，生成含有效期的加密令牌，后续请求需携带令牌验证身份，防止未授权访问；密码存储采用不可逆加密算法（如 BCrypt），避免明文泄露风险。
数据传输过程采用 HTTPS 协议加密，防止信息被拦截篡改；后台设置接口访问频率限制，抵御恶意请求与暴力攻击（如多次尝试登录、批量请求数据）。
文件上传功能需校验文件格式与大小，仅允许 PNG、JPG、PDF、DOCX 等安全格式，拒绝可执行文件（.exe 等），同时对文件名进行 MD5 加密，避免恶意文件上传与文件名冲突；敏感数据（如财务明细、身份证信息、联系方式）需加密存储，仅授权角色可查看，确保数据合规使用。
不同角色权限严格隔离，通过拦截器验证操作权限，防止越权访问（如普通学生无法查看其他社团财务数据、社团管理员无法审批其他社团活动）；系统日志记录关键操作（登录、权限变更、数据修改、文件上传等），包括操作人、操作时间、操作内容、IP 地址，供安全审计与问题追溯。
3.3.3	系统可扩展性需求
系统架构需具备灵活的可扩展性，支持功能模块的横向拓展与业务逻辑的纵向深化。核心代码采用模块化设计，提取通用工具类（数据验证、文件上传、消息推送等）与公共组件（表格展示、表单提交、弹窗提示等），减少功能扩展时的代码冗余。
数据库设计预留扩展字段，如社团表预留 “特色标签”“合作单位”“官方公众号” 字段，活动表预留 “线上直播地址”“报名表单自定义字段”“安全预案” 等，适配未来业务场景拓展（如新增线上活动、社团合作功能）。
支持与校园其他信息系统对接，如教务系统（获取学生课程信息，避免活动时间冲突）、校园卡系统（实现活动缴费与经费结算）、人脸识别系统（优化活动签到效率，支持人脸签到）、消息推送系统（集成校园公众号、短信通知）等。
前端支持多终端适配，可快速拓展小程序、APP 等移动端形态，满足用户随时随地操作的需求（如移动端报名活动、查看通知、签到考勤）；支持主题风格自定义，可根据高校品牌特色调整界面配色与布局。
3.4	本章小结
本章围绕学生社团信息整理系统的核心需求展开，明确了系统的业务定位与功能边界。通过分析四大角色的核心诉求，划分了后台管理与前端用户两大模块的功能范围，细化了用户管理、社团管理、活动管理、资源管理等核心功能的具体要求，并补充了系统整体用例图与管理模块功能结构图，直观呈现角色权限分配与功能模块关联。同时，从性能、安全、可扩展性三个维度明确了非功能性需求，为系统设计与开发提供了清晰的依据。
系统功能性需求聚焦 “多角色适配、全流程覆盖、精细化管理”，非功能性需求则保障 “高效稳定、安全可靠、灵活拓展”，两者结合既解决传统社团管理中信息分散、流程脱节、效率低下等痛点，又为未来业务拓展预留了空间，确保系统能长期适配高校智慧校园建设的发展需求。
























4	社团信息整理系统总体设计
系统采用前后端分离架构，基于 Vue+Spring Boot 技术栈构建，划分为后台管理与前端用户两大模块，通过四大核心角色权限管控实现社团管理全流程数字化，配套完整数据库设计与流程规范保障系统高效运行。
4.1	系统的体系结构
系统遵循 “高内聚、低耦合” 原则，采用分层架构与模块化设计结合的方式，整体分为客户端层、网关层、应用服务层、数据层四个核心层级，各层级职责明确、协同联动。

4.2	系统的模块划分和功能设计
系统分为后台管理模块与前端用户模块，各模块下按业务流程拆解为细分功能单元，明确输入输出、操作权限与业务规则。
4.2.1	后台管理模块功能设计
后台管理模块的设计旨在为超级管理员和社团管理员提供高度灵活性的数据管理，通过集成的界面与前端系统交互，实现了对数据、用户和系统的管理，确保了数据的安全、高效管理。
（1）用户登录与权限管理
后台管理系统中的登录功能面向超级管理员和社团管理员，该模块主要用于登录后台系统和验证用户角色权限。
•	    用户登录：用户需在界面的提示框内输入用户名和密码，确认提交后，前端页面会发送 HTTP 请求，同时将输入的账号信息传递至后端，系统会根据数据库信息校验所提交的账号信息。校验通过时，系统会生成唯一的 token 令牌作为当前用户的身份标识，并使用 Redis 缓存当前用户账户信息。
•	    权限验证：当用户通过 Web 端登录时，服务应用层的权限管理模块会基于用户表校验其账号密码与角色信息。若为超级管理员，生成携带 “super_admin” 角色标识的含有效期和签名的 JWT 令牌；若为社团管理员，生成携带 “club_admin” 角色标识的 JWT 令牌。后续请求中，客户端在 HTTP 头附加该令牌，服务端通过 SpringBoot 拦截器验证令牌有效性、解析角色信息，确认权限后放行至对应后台功能，若角色非法或令牌过期则返回权限错误，确保后台管理功能的安全性。
（2）用户信息管理
用户信息管理的最终目的是支持超级管理员全面管理所有用户的个人信息，包括学生、社团管理员、指导老师的身份信息、联系方式、所属院系、角色权限等关键信息，并能够对其进行核实、增删改查及角色分配操作，确保用户信息的有效性与权限管控的准确性。同时支持按账号、姓名、角色、院系等条件精准查询，以及用户数据的批量导出，便于管理统计。
（3）社团管理
社团管理模块的核心功能是支持超级管理员和社团管理员对社团信息进行全生命周期管理。
超级管理员：负责社团注册审批，查看社团提交的资质材料（社团章程、发起人信息、指导老师证明等），批准或驳回注册请求并填写审核意见；对已注册社团进行分类管理、星级评定、评优申报审核及注销备案审核；查看全校社团运营数据，包括成员规模、活动场次、活跃度等。
社团管理员：维护社团基础信息，编辑社团简介、更新风采照片与活动成果、管理社团标签；提交社团注销备案申请，说明注销原因并上传相关材料；查看本社团成员构成与活跃度统计数据。
（4）活动管理
活动管理模块支持管理员对活动全流程进行管控，超级管理员可查看全校所有活动的报备情况与执行状态，审核活动方案、协调资源冲突、干预违规活动；社团管理员可发布活动信息、设置报名条件、上传策划案与海报、提交活动报备申请，活动结束后上传总结与成果资料完成归档。该页面以列表形式展示活动主题、时间、地点、报名人数、审批状态等信息，支持按活动类型、时间、状态等条件筛选查询，点击查看按钮可获取活动详情、报名名单、考勤数据等信息。
（5）资源与数据统计管理
资源管理：超级管理员负责场地、器材等资源的信息维护、使用调度与申请审批，协调资源冲突；社团管理员可提交资源使用申请，跟踪审批进度。同时支持社团经费收支明细管理，记录会费收入、活动支出、赞助款项，要求上传支出凭证，支持财务数据导出与合规性核查。
数据统计管理：自动采集社团数量、成员规模、活动场次、参与率、成员活跃度等核心指标，生成可视化报表与排行榜，支持按周 / 月 / 学期筛选查询，数据可导出为 Excel 格式，为管理决策提供数据支撑。
4.2.2	前端用户模块功能设计
前端用户模块通过功能模块实现移动端、Web 端与后台系统之间的数据交互。用户通过终端设备操作系统，系统发送 HTTP 请求，后台系统接收后执行相应路由指令，实现与数据库的数据交换，最终处理结果以提示信息或页面展示的方式反馈给用户。前端用户模块的主要功能分为用户登录注册、个人信息管理、社团与活动交互、指导老师专属功能等。
（1）用户登录注册功能
注册功能是任何用户访问系统的必要前提，依托此功能可达成身份登记。用户需填写学号、密码、姓名、所属院系、联系方式等个人账号信息，点击确认后，系统将信息传输至后端，由后台进行信息验证（校验学号唯一性、信息完整性）。验证成功后，用户凭借唯一的身份标识进入系统前端界面，完成后续操作；若信息重复或不完整，返回错误提示并引导补充修改。
登录功能为用户提供进入系统首页的必要流程，用户在登录界面输入注册后获得的用户名和密码，校验通过后即可享受后续服务。操作时，系统界面提供对应提示，输入完毕点击确认后，系统根据数据库信息校验输入数据，校验成功则登入系统主页面；若数据库无法查询到相应字段即校验失败，给出用户名或密码不正确的错误提示并保持登录态锁定。用户登录注册功能流程图如图 4.2 所示。

（2）实体之间的联系
实体之间的联系在信息处理和数据库设计中扮演核心角色，明确实体间关系是构建数据模型和索引的关键，以下对本系统设计中各实体之间的联系做具体分析：
1. 超级管理员与用户、社团、活动、资源申请：均为一对多关系。超级管理员可管理所有用户、审核所有社团注册与活动报备、审批所有资源申请。
2.社团管理员与社团、活动、社团成员：均为一对多关系。一个社团管理员对应一个社团，可发起多个活动、管理该社团的所有成员。
3.指导老师与社团：为一对多关系。一个指导老师可指导多个社团，一个社团对应一个指导老师。
4.用户（学生）与社团成员、活动报名、评价：均为一对多关系。一个学生可加入多个社团、报名多个活动、提交多个评价；一个社团成员、活动报名、评价对应一个学生。
5.社团与活动、社团成员、资源申请、通知：均为一对多关系。一个社团可发起多个活动、包含多个成员、提交多个资源申请、发布多个通知；一个活动、社团成员、资源申请、通知对应一个社团。
6.资源与资源申请：为一对多关系。一个资源可接收多个申请，一个申请对应一个资源。
7.用户与通知：为多对多关系。多个用户可接收同一个通知，一个用户可接收多个通知，通过通知表的接收范围字段关联。

4.4	本章小结
本章主要对学生社团信息整理系统的两大组成部分和数据库进行了总体设计介绍。在体系结构设计中，系统主要由后台管理系统和前端用户模块组成，基于三层架构实现请求处理、业务逻辑与数据操作的解耦，符合 MVC 设计模式，保障了系统的扩展性与维护性。在模块划分和功能设计中，后台管理模块涵盖用户登录与权限管理、用户信息管理、社团管理、活动管理、资源与数据统计管理等功能；前端用户模块囊括用户登录注册、个人信息管理、社团与活动交互、指导老师专属功能等，实现了多角色的差异化需求覆盖。在数据库概念设计中，明确了核心实体、属性及实体间关联关系，完成了实体关系模型构建，为后续数据库表设计与系统开发奠定了基础。
5	社团信息管理系统实现
本章基于系统总体设计方案与论文中明确的实体属性，完成前后端核心功能的开发与集成。系统采用前后端分离架构，后端以 Spring Boot 为核心搭建服务体系，前端基于 Vue 3.x 构建交互界面，通过 MySQL 存储全量实体数据、Redis 缓存热点信息，最终实现覆盖四大角色的社团管理全流程数字化。
5.1	系统技术分析
学生社团信息整理系统通过 SpringBoot+Vue 技术搭建前后端分离架构的核心框架，后文将从服务端架构设计与客户端交互逻辑两个维度，详细阐述系统核心技术栈。
本系统依托 SpringBoot 技术框架实现后端核心服务，基于 MVC 三层架构封装社团管理、活动组织、权限控制等核心业务逻辑，通过 JWT 权限验证技术保障多角色登录可靠性与未授权操作防护；借助 MyBatis 框架完成高效数据库交互，通过 MySQL 建立用户、社团、活动、资源等全量数据表，配合 Redis 技术缓存热门数据以提升查询效率与系统性能，利用 MyBatis 动态 SQL 实现复杂关联查询与数据更新，支撑社团审批、活动报名、成员管理等预设功能；前端通过 Vue 框架结合 Element Plus 组件库构建响应式交互界面，借助 Axios 调用 SpringBoot 后端提供的 RESTful API 实现数据交互，通过 Vue Router 实现路由管控与权限拦截，依托 Pinia 管理全局状态，适配 PC 端与移动端多终端使用场景，实现社团浏览、活动报名、个人信息维护等前端功能。

5.3	系统后台管理系统的实现
基于前面的功能设计分析，后台管理模块聚焦超级管理员与社团管理员的核心管理场景，通过 Spring Boot 三层架构（Controller-Service-Mapper）实现权限管控、用户管理、社团管理、活动管理等核心功能。以下结合 UML 类图与核心逻辑，详细阐述各功能模块的实现细节。
5.3.1	超级管理员信息管理的实现
超级管理员通过后台登录页输入账号密码，前端发送 POST 请求至 /api/super-admin/login 接口。后端SuperAdminController 接收请求后，调用 SuperAdminService 的 verifyLogin 方法，通过 SuperAdminMapper 查询数据库，采用 BCrypt 算法比对密码合法性。校验通过后，生成携带 super_admin 角色标识、有效期 2 小时的 JWT 令牌，存入 Redis 缓存。后续请求通过 HTTP 头携带令牌，JwtInterceptor 拦截验证有效性与角色权限，仅允许超级管理员执行用户管理、社团审批等核心操作。管理员信息管理功能类图如图5.1所示。
 
图5.1  超级管理员信息管理功能类图

5.3.2	社团管理员信息管理的实现
社团管理员登录后，通过 /api/club-admin/info 接口获取关联社团信息（基于 manage_club_id 关联）。编辑个人信息时，前端发送 PUT 请求至 /api/club-admin/update，ClubAdminController 调用服务层校验权限（仅允许编辑自身信息）。ClubAdminService 通过 ClubAdminMapper 更新数据库记录，同步更新社团关联信息（如负责人联系方式）。
支持查询社团运营数据（成员数、活动数），通过关联查询 club 与 activity 表返回统计结果。社团管理员信息管理功能类图如图5.2所示。
 
图5.2  社团管理员信息管理功能类图
5.3.3	学生信息管理实现
学生通过学号 + 密码登录，前端发送请求至 /api/student/login，StudentController 调用服务层校验身份。个人信息维护时，通过 /api/student/update 接口提交修改数据，StudentService 校验字段合法性（学号不可修改）。StudentMapper 执行数据库更新操作，同步更新关联的 club_member 表中成员联系方式等关联信息。支持查询个人参与记录（已加入社团、已报名活动），通过多表关联查询返回聚合结果。学生信息管理功能类图如图5.3所示。
 
图5.3  学生信息管理功能类图

5.3.4	指导老师信息管理实现
指导老师通过工号 + 密码登录，前端请求至 /api/teacher/login，TeacherController 调用服务层校验。登录后通过 /api/teacher/guided-clubs 接口获取所指导社团列表，TeacherService 关联查询 club 表中 guide_club_id 匹配数据。编辑个人信息（如专业方向、联系方式）时，通过 /api/teacher/update 接口提交，TeacherMapper 执行更新操作，同步更新社团表中指导老师关联信息。指导老师信息管理功能类图如图5.4所示。

超级管理员通过 /api/club-category/add 接口新增分类，提交分类名称、父分类 ID、描述等数据。ClubCategoryController 调用服务层校验数据合法性（父分类需存在），ClubCategoryService 通过 Mapper 插入数据库。支持分类编辑、删除（删除前校验是否有关联社团）、列表查询，通过动态 SQL 实现多条件筛选与分页。社团分类信息管理功能类图如图5.5所示。

5.3.6	活动管理功能实现
社团管理员通过 /api/activity/add 接口发布活动，提交主题、时间、地点、报名条件等数据，上传策划案与海报。ActivityController 调用服务层校验数据完整性，ActivityService 生成活动唯一编号，通过 Mapper 插入 activity 表（状态为 “待审核”）。超级管理员通过 /api/activity/approve 接口审核，更新活动状态为 “已通过” 或 “已驳回”，并推送通知给社团管理员。活动执行后，支持报名名单导出、考勤记录录入、活动总结归档，通过多表关联实现全流程数据追踪。活动管理功能类图如图5.6所示。

学生参与活动后，通过 /api/evaluation/submit 接口提交评价，包括评分（1-5 星）、文字内容、现场照片（可选）。EvaluationController 调用服务层校验评价时效（活动结束后 7 天内），EvaluationService 通过 Mapper 插入 evaluation 表。社团管理员通过 /api/evaluation/club-list 接口查看本社团评价，支持回复评价内容。超级管理员可通过 /api/evaluation/all-list 接口查询所有评价，对违规评价执行删除操作。评价信息管理功能类图如图5.7所示。

5.4	前端用户模块的实现
前端用户模块基于 Vue 3.x + Element Plus 开发，适配 PC 端与移动端，通过 Axios 与后端 Spring Boot 服务异步通信，实现登录注册、个人信息管理、社团与活动交互等核心功能。结合论文 “4.2.2 前端用户模块功能设计” 的需求定义，模块严格遵循 “多角色适配、全流程覆盖” 的设计原则，借助 Vue 生态工具链快速构建交互界面，配合后端 RESTful API 完成数据传输，满足学生、社团管理员、指导老师的差异化操作需求，实现社团管理全流程数字化。
5.4.1	用户登录注册功能的实现
用户的登录流程由前端请求、后端验证和状态管理三个阶段组成：用户填写完学号 / 工号和密码后调用 Login.vue 组件提交，登录请求经 JWT 拦截器转发至后端 WebController，服务层 UserService 通过 UserMapper 执行数据库查询操作，校验用户账号是否存在及密码是否正确（密码通过 BCrypt 算法加密比对）。若账号或密码错误则显示 “账号或密码错误” 提示，若验证通过则生成包含用户 ID 和角色（student/club_admin/teacher/super_admin）的 JWT 令牌，并查询用户关联的社团信息（学生查已加入社团、老师查指导社团）更新身份属性。最后返回令牌及验证结果至前端，存储令牌并跳转至对应角色的主界面（学生跳转首页、管理员跳转管理后台）。

5.4.3	社团与活动交互功能实现
社团与活动交互功能实现流程如下：用户访问社团列表页面后，前端通过 load () 方法调用接口，获取社团分类、热门社团及推荐活动数据（按活跃度、创建时间排序）并加载至页面；点击社团详情页可查看社团简介、成员构成、历史活动、指导老师等信息，提交入社申请时 submitApply () 会对申请理由等表单数据进行校验并发送请求，后端经 JWT 拦截器验证权限后，由控制器接收并通过 ClubService 处理业务逻辑，通过 ClubMemberMapper 操作数据库完成申请记录存储（状态为 “待审核”）；活动报名流程类似，用户选择活动后提交报名信息，后端校验报名条件（年级、专业限制）和人数上限，通过后创建报名记录，处理结果经由统一状态码返回前端，成功后刷新页面数据并提示操作结果，失败则显示具体错误信息。
